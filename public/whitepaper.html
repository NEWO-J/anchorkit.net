Technical White Paper

AnchorKit: Hardware-Rooted Photo Provenance on the Solana Blockchain

Author: Jonah Owen Date: April 2026 Version: 1.0 Programme ID: HF869x3899LMrFXd2U5sa7zECYXjnnhrzqwR4pPiyWwb
Abstract

Determining whether a digital photograph is authentic has become one of the more consequential unsolved problems in information security. Generative image models have progressed to the point where distinguishing a real photograph from a synthetic one is beyond reliable human capability, and automated content-based detectors fare only marginally better in adversarial settings. This paper describes AnchorKit, a photo provenance system for Android devices that approaches the problem from a fundamentally different direction: rather than inspecting the image for artefacts of synthesis, AnchorKit cryptographically binds a hash of the raw captured image to the specific hardware that produced it, at a specific point in time, and records that commitment to an immutable, decentralised ledger where it can be verified by anyone, forever, without contacting AnchorKit's servers.

The system achieves this through four coordinated mechanisms: an Android SDK that generates an EC P-256 signing key inside the device's TEE or StrongBox and signs a SHA-256 image hash with a server-issued single-use nonce; a Python backend that validates the full X.509 attestation chain rooted at Google's attestation CA and enforces verifiedBootState = Verified, origin = GENERATED, and hardware security level; a Rust Merkle tree builder that aggregates daily submissions into a single root; and a Solana on-chain programme (live on mainnet) that anchors that root immutably. Verification requires only SHA-256 arithmetic and a public Solana RPC call.

§1 Introduction

1.1 The Problem with Content-Based Detection

The default response to the proliferation of AI-generated imagery has been to develop detectors: classifiers trained to distinguish synthetic images from real ones based on statistical artefacts — checkerboard patterns from deconvolution, unnatural frequency spectra, spatial inconsistencies in lighting, and similar signals. This approach is fundamentally reactive. Every detector trained on artefacts from a given generation of models can be rendered ineffective by the next generation, which is explicitly trained — or trivially fine-tuned — to suppress those artefacts. This is not a solvable engineering problem; it is an adversarial arms race in which the defender starts at a structural disadvantage.

A more durable approach shifts the burden of proof. Instead of asking "does this image look real?" ask "can this image prove it was captured in the physical world on specific hardware, at a specific time, with a commitment recorded before anyone could have known how to manipulate it?" This is the question AnchorKit answers.

1.2 Why Hardware Roots Matter

Software-only signing schemes — including the widely deployed C2PA standard — allow any credential holder to sign any content at any time. A C2PA credential issued to an application running on a developer's laptop can produce a signed manifest for a generated image with the same cryptographic structure as a manifest produced by a real camera. The credential does not encode the physical reality of the capture event; it encodes only the identity of whoever held the signing key.

Hardware attestation is qualitatively different. When a key is generated inside a Trusted Execution Environment or StrongBox secure element on an Android device, the key material never appears in main processor memory and cannot be extracted by any API available to application or system software. Android Key Attestation (available from Android 7.0 onward) provides cryptographic proof of these properties: a certificate chain signed by a Google-operated attestation CA, whose leaf certificate carries a structured extension encoding the key's security level, its origin (generated vs. imported), and the verified boot state of the device at key generation time.

1.3 Why Blockchain Anchoring Matters

A valid hardware attestation establishes that a signing key genuinely lives in a TEE or StrongBox on a verified-boot device. But attestation alone does not establish when an image was captured. AnchorKit addresses this through a server-issued, single-use nonce with a five-minute TTL: the nonce must be bound into the signed payload and consumed before the submission is accepted. This makes retroactive attestation computationally infeasible.

The blockchain layer provides immutability and public accessibility of the commitment. Once a Merkle root is posted to Solana, no party — including AnchorKit — can modify or retract it. A verifier in a future proceeding can retrieve the root from any Solana RPC endpoint and compute the proof arithmetic locally, without depending on AnchorKit's availability or integrity.

§2 Background

2.1 Android Hardware Attestation

Android's Keystore system provides access to cryptographic keys stored inside a hardware-backed environment. Two types of secure hardware are relevant. A Trusted Execution Environment (TEE) is an isolated execution environment, typically implemented using Arm TrustZone (Arm Limited, 2020), that runs alongside the main OS but is cryptographically separated from it. Keys stored in the TEE are inaccessible to the normal world even if the Android OS is fully compromised. A StrongBox is a dedicated security chip — a physically separate integrated circuit with its own processor, memory, and power supply — that meets the Android StrongBox Keymaster HAL specification and provides stronger physical isolation guarantees.

Android Key Attestation (Google, 2023b) lets applications obtain an X.509 certificate chain for a hardware-backed key, signed by a Google-operated attestation root CA. The leaf certificate carries a custom extension under OID 1.3.6.1.4.1.11129.2.1.17 that encodes the attestationSecurityLevel (Software=0, TrustedEnvironment=1, StrongBox=2), the hardwareEnforced.origin (GENERATED=0, DERIVED=1, IMPORTED=2, UNKNOWN=3), and the hardwareEnforced.rootOfTrust.verifiedBootState (Verified=0, SelfSigned=1, Unverified=2, Failed=3). These fields are populated inside the TEE or StrongBox itself and cannot be forged by application or OS software. Cooijmans et al. (2014) analysed Android key storage security at an earlier stage; the attestation mechanism described here postdates and addresses the vulnerabilities they identified.

The verifiedBootState field is particularly significant. Android Verified Boot validates the integrity of the OS image during startup using a chain of trust rooted in a hardware-stored key. A device running an unmodified, Google-verified OS carries verifiedBootState = Verified; an unlocked bootloader or tampered OS will carry a different value. Because boot state is recorded at key-generation time, it reflects the device's integrity when the attested key was created.

2.2 Merkle Hash Trees

A Merkle tree (Merkle, 1987) is a binary hash tree in which leaf nodes contain data values and each internal node contains the hash of the concatenation of its two children. For n leaf values, the tree produces a root hash R that commits to the entire set. Membership of a specific leaf h can be proved with a proof path of size O(log n) — the sibling hashes at each level from leaf to root. Certificate Transparency (Laurie et al., 2013) deployed this construction at internet scale for TLS certificates. AnchorKit applies it to daily batches of image hash submissions, with the root anchored on a public blockchain.

2.3 Blockchain Timestamping

Gipp et al. (2015) proposed using Bitcoin's blockchain as a decentralised timestamping service, exploiting its append-only properties to establish document existence at a particular time. AnchorKit applies this principle to image provenance: once a Merkle root is posted to Solana, its existence is permanently recorded and independently verifiable without trusting AnchorKit.

2.4 Image Forgery Detection

Farid (2009) surveyed image forgery detection, cataloguing copy-move, splicing, and statistical anomaly methods. Tolosana et al. (2020) extended this to the deepfake era. Both surveys converge on the same structural observation: detectors relying on artefacts of a specific synthesis generation are reliably circumvented by the next generation. AnchorKit's provenance-by-attestation approach sidesteps this entirely: if a submission cannot produce a valid hardware attestation, it is rejected regardless of whether the image content appears authentic.

§3 System Architecture

AnchorKit consists of four components:

  • Android SDK (io.anchorkit.sdk) — Kotlin library responsible for device integrity checks, image capture and hashing, hardware key management, attestation certificate generation, and signed submission. Available at net.anchorkit:sdk:1.0.1.
  • Python Backend (FastAPI on AWS) — issues nonces, validates submissions, stores records in DynamoDB, runs the nightly batch process.
  • Rust Merkle Tree Builder (merkle_tree_builder_rs) — a compiled PyO3 extension module called by the nightly batch to construct the tree from each day's submissions.
  • Solana On-Chain Programme (HF869x3899LMrFXd2U5sa7zECYXjnnhrzqwR4pPiyWwb) — built with the Anchor framework, stores daily Merkle roots in programme-derived accounts. Live on Solana mainnet.

The data flow is:

  1. At capture time, the SDK hashes raw image bytes in memory, requests a nonce, signs hash:nonce:metadataHash with the hardware key, and submits with the attestation certificate chain.
  2. The backend runs eight sequential validation checks and writes the record to DynamoDB with attestation_verified = True.
  3. Each night at midnight UTC, records are sorted by hash_id, aggregated into a Merkle tree, and the root is posted to Solana.
  4. At any future time, a verifier requests a PortableProof bundle and verifies it offline using SHA-256 arithmetic and any public Solana RPC endpoint.

§4 Phase I — Client-Side Capture & Attestation

4.1 Device Integrity Checks

Before initiating any capture workflow, the SDK's DeviceIntegrity object performs client-side checks as defence-in-depth. These checks are explicitly documented in the source code as not being security controls — a sufficiently privileged attacker can bypass any of them. The authoritative enforcement is server-side attestation. The checks fail fast with a meaningful error and raise the bar against automated attacks.

  1. Build tag inspection: Build.TAGS is examined for "test-keys", indicating a non-production OS build typical of custom ROMs and unlocked bootloaders.
  2. Emulator detection: Multiple signals are checked — Build.FINGERPRINT for tokens like "generic" and "google_sdk"; Build.HARDWARE for goldfish/ranchu; Build.PRODUCT for "sdk" and "vbox"; Build.MODEL for emulator strings; and Build.MANUFACTURER for Genymotion.
  3. Root binary detection: Known paths are tested for existence: /system/xbin/su, /data/adb/magisk.db, /sbin/magisk, and others.
  4. System partition writability: /system is tested for write access. On stock Android this partition is always read-only; writability indicates a rooted device.

Certificate pinning is implemented via SPKI SHA-256 pinning directly on HttpsURLConnection — not via network_security_config.xml, which can be overridden by installing a custom system CA. This implementation is not overridable by device-level configuration.

4.2 In-Memory Hashing

Raw image bytes delivered by CameraX are hashed with SHA-256 (NIST, 2015a) before any write to the device filesystem. This ensures the commitment corresponds to the camera sensor output, not to any subsequent encoding or processing applied during filesystem write. The hash is encoded as a 64-character lowercase hexadecimal string. EXIF TAG_DATETIME_ORIGINAL is extracted as auxiliary metadata but is not relied upon for security properties — timing integrity comes exclusively from the nonce TTL.

4.3 Hardware Key Generation and Management

On first use, EnclaveAttestation.sign() generates an EC P-256 key pair (secp256r1) inside the Android Keystore under alias anchorkit_attestation_key. The KeyGenParameterSpec sets PURPOSE_SIGN, specifies DIGEST_SHA256, and passes an attestation challenge (a UUID persisted in SharedPreferences). On API 28+, StrongBox is attempted first; StrongBoxUnavailableException causes fallback to TEE. The Android Keystore API exposes no mechanism to export private key material from a hardware-backed key.

A 30-day maximum key age is enforced (KEY_MAX_AGE_MS = 30 × 24 × 3600 × 1000 ms). This addresses a subtle attack: a key generated when the device was in a clean verified-boot state carries verifiedBootState = Verified even if the device is subsequently rooted. Periodic regeneration limits the window during which a compromised device can present a stale clean certificate. The SDK also checks for certificate chain expiry; if any certificate in the stored chain has expired, regeneration is forced to obtain a fresh chain from the current intermediate CA.

4.4 Nonce Retrieval and Signed Payload Construction

Prior to submission, the SDK calls GET /api/attestation-challenge. The server generates a 256-bit random nonce, stores it in DynamoDB with a five-minute TTL, and returns it. The five-minute window prevents retroactive attestation: an image captured now cannot be submitted later under a nonce issued for a different moment.

The SDK constructs the signed payload as the UTF-8 encoding of:

"${hash.lowercase()}:${nonce}:${metadataHash}"

where metadataHash is the lowercase hex SHA-256 of the canonical metadata string — key-value pairs sorted lexicographically by key, joined as "key1=value1,key2=value2". Binding the metadata hash into the signed payload prevents a network-layer attacker from silently modifying submission fields (timestamp, image dimensions) in transit — any such modification invalidates the signature.

The payload is signed using ECDSA with SHA-256 (SHA256withECDSA) under the hardware-backed key. The certificate chain is serialised as Base64(JSON([Base64(cert_leaf), Base64(cert_intermediate), ..., Base64(cert_root)])).

§5 Phase II — Server-Side Validation

The endpoint POST /api/submit-hash executes eight validation steps in strict sequence. Failure at any step terminates the request with a 400 or 403 response.

  1. Hash format — The submitted hash is validated against /^[0-9a-f]{64}$/. This rejects any value that is not a valid SHA-256 hex digest.
  2. API key — The X-API-Key header is validated against the set of keys for the requesting account.
  3. Metadata bounds — At most 20 keys; each key ≤ 64 characters; each value ≤ 512 bytes. This constrains the surface area of the metadata that becomes part of the signed payload and prevents denial-of-service through oversized submissions.
  4. Atomic nonce consumption — The nonce is consumed via a DynamoDB conditional update: SET used = True WHERE used = False AND expires_at > now(). DynamoDB conditional updates are atomic at the item level, meaning concurrent requests carrying the same nonce will result in exactly one success. The nonce is consumed before signature verification to prevent a race-window replay attack in which an attacker intercepts a valid attestation from an in-flight request.
  5. Certificate chain validation — Each certificate is parsed from DER, checked for temporal validity (not_valid_before / not_valid_after against current UTC), and the cryptographic chain is verified (each certificate's signature checked against its parent's public key). The root must be self-signed and its SHA-256 fingerprint must appear in the operator-pinned set of Google attestation CA fingerprints.
  6. Enclave signature verification — The leaf certificate's public key is used to verify the ECDSA/SHA-256 signature over the canonical message. The metadataHash is reconstructed server-side from submitted fields using the same lexicographic key-sort procedure as the client. A valid signature proves: (a) the attested device signed this specific hash, (b) this specific nonce was incorporated, and (c) metadata fields were not modified in transit.
  7. Attestation extension parsing and enforcement — The extension under OID 1.3.6.1.4.1.11129.2.1.17 is extracted from the leaf certificate and parsed by a custom DER reader with full bounds checking. Three fields are enforced:
    • attestationSecurityLevel must be TrustedEnvironment (1) or StrongBox (2). Software (0) is rejected.
    • hardwareEnforced.origin must be GENERATED (0). IMPORTED, DERIVED, UNKNOWN, and absent are all rejected. Treating an absent field as a pass would allow an omission-bypass attack where an attacker crafts a certificate that simply omits the field.
    • hardwareEnforced.rootOfTrust.verifiedBootState must be Verified (0). All other values and absent are rejected for the same reason.
  8. Record storage — The record is written to the active DynamoDB hot table with attestation_verified = True, a sequential hash_id (assigned via atomic counter), the submission day, timestamp, and the leaf certificate fingerprint. Hot records carry a 72-hour TTL. The backend maintains three DynamoDB tables (A, B, C) in rotation — A and B alternate as the active hot table, swapping at midnight UTC; C is an emergency failover.

§6 Phase III — Merkle Tree & Solana Anchoring

6.1 Nightly Batch Process

At midnight UTC the backend swaps the active DynamoDB table, filters records for the preceding day by day == yesterday, and sorts them by hash_id ascending, producing a deterministic ordered sequence [h₁, h₂, …, hₙ]. The full record set is archived to S3 as archives/{date}.jsonl.gz. A sharded lookup index is written to shards/{p2}/{p4}.jsonl.gz (partitioned by the first two and four hex characters of each hash), enabling O(1) lookup during proof generation.

6.2 Merkle Tree Construction

The Rust extension module merkle_tree_builder_rs builds the tree as follows:

  • Leaves: the hash_id-ordered SHA-256 hex strings as submitted — 64-character lowercase hex.
  • Internal nodes: each internal node is the SHA-256 of the UTF-8 concatenation of its two children's hex strings:
node = SHA-256(left_hex_string ∥ right_hex_string)

This means the input to SHA-256 is the concatenation of two 64-character hex strings as UTF-8 bytes (128 bytes total), producing another 32-byte hash encoded as a 64-character hex string. Verification is straightforward in any language with string concatenation and SHA-256.

  • Odd levels: when a level has an odd number of nodes, the last is duplicated before computing parents (standard convention per Merkle, 1987).
  • Root: the process repeats until one node remains. For n leaves, the tree has ⌈log₂n⌉ + 1 levels and any proof has ⌈log₂n⌉ elements.

6.3 Solana Programme and On-Chain Data Structure

The AnchorKit Solana programme (deployed at HF869x3899LMrFXd2U5sa7zECYXjnnhrzqwR4pPiyWwb) stores Merkle roots in programme-derived accounts (PDAs) whose addresses are deterministically derived from seeds [b"merkle_registry", chunk_index.to_le_bytes()]. PDAs have no private key and can only be written by the programme itself.

Each MerkleRootRegistry account stores up to 10 MerkleRootEntry records, where each entry contains:

  • merkle_root: 32 bytes
  • date: UTF-8 string, e.g. "2026-04-07"
  • timestamp: i64 Unix timestamp

The programme enforces duplicate-date prevention — before inserting a new entry it checks that no existing entry in the current chunk carries the same date string, rejecting duplicates with AnchorKitError::DuplicateDate. When a chunk reaches capacity, create_next_registry allocates chunk N+1 and links chunk N's next_chunk field to it, forming a singly-linked list traversable from chunk 0. The programme enforces sequential chunk creation: new_chunk_index must equal old.chunk_index + 1 and the old chunk must be full before a new one is created.

The borsh serialisation format used by Anchor is deterministic and language-independent, allowing any verifier to parse raw account data using any borsh library.

6.4 Transaction Cost Model

Each add_merkle_root transaction on Solana mainnet costs approximately 5,000 lamports (~$0.0007 at early 2026 prices). Since one transaction anchors an entire day's submissions, the per-record cost approaches zero at scale:

cost_per_record ≈ $0.0007 / n → ~$0.0000007 at n = 1,000 submissions/day

This is the economic advantage of batched Merkle anchoring over per-record blockchain transactions.

§7 Verification

7.1 The PortableProof Bundle

After the nightly batch, a PortableProof is available via GET /api/verify-hash/{hash}:

PortableProof {
  schema_version:      Int     // Always 1
  hash:                String  // SHA-256 hex of the image
  day:                 String  // Submission date (YYYY-MM-DD)
  timestamp:           Long    // Submission epoch seconds
  hash_id:             Int     // 0-indexed position in daily batch
  merkle_root:         String  // Merkle root anchored on-chain (hex)
  merkle_proof:        List<List<String>> // [[sibling, "left"|"right"], ...]
  solana_program:      String  // On-chain programme ID
  solana_registry_pda: String? // Base-58 PDA address
  solana_chunk_index:  Int?    // Chunk index for PDA derivation
  solana_tx:           String? // Transaction signature (audit trail)
  chain:               String  // "mainnet" or "devnet"
}

Once obtained, the bundle should be stored by the application — it remains verifiable without any AnchorKit server contact.

7.2 Offline Verification Algorithm

verifyLocally(proof) performs two independent checks:

Check 1 — Merkle proof recomputation

Let h be the image hash and π = [(s₁, d₁), …, (sₖ, dₖ)] the proof path:

v₀ = h For j = 1 to k: if d_j == "left": v_j = SHA-256(s_j ∥ v_{j-1}) // sibling is left child else: v_j = SHA-256(v_{j-1} ∥ s_j) // sibling is right child Claimed root = v_k

∥ denotes UTF-8 string concatenation of the two 64-character hex strings. SHA-256 is applied to the resulting 128 UTF-8 bytes.

Check 2 — On-chain root confirmation

The verifier re-derives the PDA address from solana_program and solana_chunk_index using seeds [b"merkle_registry", chunk_index.to_le_bytes()], calls getAccountInfo on any public Solana RPC endpoint, deserialises the borsh-encoded MerkleRootRegistry struct, locates the entry whose date field matches proof.day, and checks that its merkle_root bytes equal the hex root vₖ computed in Check 1.

Zero Trust

This verification requires no contact with AnchorKit's servers, no trust in AnchorKit, and no cryptographic operations beyond SHA-256 and byte comparison. The solana_tx field is an audit-trail reference only — verification reads account state, not transaction history.

§8 Security Analysis

8.1 Replay Attacks

An intercepted submission payload cannot be reused. The nonce is consumed atomically before any other validation step; subsequent attempts to submit the same nonce see used = True and are rejected immediately. The five-minute TTL further limits exposure. The nonce is cryptographically bound into the signed payload, so substituting a fresh nonce invalidates the signature.

8.2 Software Attestation Bypass

An emulator or debug OS generates keys with attestationSecurityLevel = Software (0). This value is rejected. There is no mechanism by which software running in normal execution mode can produce a certificate claiming TEE or StrongBox security level — this property is enforced by the hardware and TEE firmware, not by the OS.

8.3 Imported Key Attack

A key generated externally and imported into the Keystore carries hardwareEnforced.origin = IMPORTED (2). AnchorKit rejects any origin value other than GENERATED (0), ensuring the signing key was created inside secure hardware and that no party has possessed the private key bytes.

8.4 Rooted or Modified Device

An unlocked bootloader or modified OS produces verifiedBootState ∈ {SelfSigned, Unverified, Failed}, all rejected. Even if an adversary keeps the TEE intact while rooting the device, the attestation extension will expose the compromised boot state. The 30-day key rotation further limits the window during which a key generated on a previously clean device can be used after subsequent rooting.

8.5 Certificate Forgery

A self-signed or third-party-rooted chain is rejected at the root fingerprint check. Forging a chain rooted at Google's attestation CA requires breaking ECDSA at the 128-bit security level (NIST, 2023), outside the scope of this threat model.

8.6 Retroactive Signing

The server-issued nonce has a five-minute TTL and is single-use. An adversary cannot obtain a nonce for a past moment — they can only request a nonce for now and submit within five minutes. Past blockchain records are immutable regardless of any future key compromise.

8.7 Man-in-the-Middle

SPKI SHA-256 certificate pinning is implemented programmatically on HttpsURLConnection. Unlike network_security_config.xml pinning, it cannot be overridden by installing a custom system CA certificate. This provides resistance to corporate MitM proxies, malicious networks, and state-level interception.

8.8 Physical Attack Vectors: The Analogue Hole

AnchorKit cannot distinguish a camera pointed at a real scene from a camera pointed at a screen displaying a synthetic image. An attacker who displays a generated image on a high-quality monitor and photographs it with an AnchorKit-enabled verified-boot device would produce a valid attestation of the photograph of the display. The system correctly attests to what it can attest to — the integrity of the capture pipeline and the device — and makes no claim about the provenance of the scene.

Photo-of-photo attacks are partially mitigated by physical optics artefacts detectable in forensic review:

  • Moiré interference patterns: The camera sensor's pixel grid and the display's pixel grid interact to produce beat-frequency interference fringes when at a slight angle. Patel et al. (2015) demonstrated moiré patterns as a reliable indicator of display-capture attempts, particularly when sensor resolution and display pixel pitch are close but not identical.
  • Display colour gamut clipping: Consumer displays cannot reproduce the full range of colours captured by a sensor. Photographs of displays show colours clipped to the display's gamut, detectable by statistical analysis of the colour distribution.
  • Parallax and depth inconsistencies: A real scene has three-dimensional depth; a flat display does not. Multi-frame parallax analysis or depth estimation may reveal the characteristic flat depth profile of a display.

Video-of-video attacks face an additional practical barrier. Most displays operate at a fixed refresh rate (60 Hz or 120 Hz) and most cameras capture at a fixed frame rate (30 or 60 fps). When these rates are close but not integer multiples of each other, rolling shutter effects and temporal sampling produce characteristic beat-frequency banding across frames. Eliminating this artefact requires precise hardware-level synchronisation of camera exposure timing to display refresh — not achievable without controlling both devices simultaneously. An attacker must either tolerate visible banding in the attested video or accept an extremely narrow window of near-perfect rate matching, making the practical attack surface severely constrained.

These physical artefacts are a complementary forensic detection layer, not enforced by AnchorKit's protocol, but a meaningful practical barrier for dispute resolution.

§9 Preventing False Positives: AI-Generated & Manipulated Images

A core question for any deployment of AnchorKit: can an AI-generated or doctored image pass the full validation pipeline?

9.1 The Nonce Barrier

An AI-generated image was not captured at a specific moment in time. To pass AnchorKit's validation, a submission must include a valid, unexpired, unconsumed nonce — issued at most five minutes before the submission. An adversary cannot obtain a nonce for a past moment. If they generate an image and submit immediately, the timestamp is "now," not the timestamp of some alleged past event. EXIF metadata can be manipulated, but EXIF timestamps carry no security weight in AnchorKit's model.

9.2 The Hardware Attestation Barrier

The submission must include a valid certificate chain rooted at Google's attestation CA with verifiedBootState = Verified, origin = GENERATED, and attestationSecurityLevel ∈ {TEE, StrongBox}. No general-purpose computing environment — a laptop, a cloud VM, a modified emulator — can produce such a chain. It requires a real Android device running unmodified firmware with a TEE or StrongBox that generated the key internally.

9.3 The Signature Binding Barrier

The certificate chain certifies a specific public key. The submission must include an ECDSA signature verifiable against that public key. The private key is inside a hardware-backed secure element and cannot be extracted. An adversary who obtains the certificate chain of a legitimate device cannot produce valid signatures without physical control of that device.

9.4 The Hash Binding Barrier

A deepfake or doctored image has a different SHA-256 hash than the original. Even if an adversary could somehow obtain a valid attestation for a legitimate image, substituting the fake would invalidate the hash binding. Any modification to the image — however small — changes the hash and breaks the proof.

Summary

Producing a valid AnchorKit proof for an AI-generated or manipulated image requires simultaneously: a valid server nonce (available only for five minutes, and only from this moment); a real Android device on verified-boot firmware with hardware-backed key generation; physical control of that device's signing operations; and a way to cause the device's CameraX pipeline to produce the target hash without capturing it from the sensor. The last requirement alone necessitates modifying the camera pipeline, which changes the device's boot state and violates the second requirement.

10.1 C2PA

C2PA (Coalition for Content Provenance and Authenticity, 2022) defines a manifest format signed using X.509 credentials, supported by Adobe, Microsoft, Intel, Arm, BBC, Sony, and others. It represents a significant advance over unsigned media, but its trust model has structural limitations in adversarial settings. Signing credentials can be issued to software applications, not solely to capture hardware. A credential holder can sign any image at any time. Manifests are silently stripped by many image processing applications and social media platforms. Verification requires a centralised content credentials service. Most critically: a compromised credential can produce backdated manifests with no external anchor to detect them. AnchorKit's blockchain records are immutable, so past records survive any future key compromise.

10.2 Truepic Lens

Truepic Lens is the closest competitor in terms of hardware-attestation depth, combining TEE-based signing on Qualcomm hardware with C2PA manifest output. Key differences: Truepic issues signing certificates through their own Certificate Authority. This means the attestation chain proves that Truepic issued a certificate, not that the key genuinely lives in hardware independently of Truepic's issuance practices. AnchorKit's reliance on Google's attestation CA provides third-party verifiability without trusting AnchorKit's infrastructure. Truepic Lens also does not anchor to a public blockchain — verification requires Truepic's centralised service.

10.3 IPTC Photo Metadata

IPTC Photo Metadata (IPTC, 2021) provides no cryptographic integrity guarantees. Fields can be modified by any application, stripped during processing, and there is no signing scheme. IPTC is useful for attribution and rights management but unsuitable for adversarial provenance verification.

10.4 Numbers Protocol

Numbers Protocol (2022) posts image hashes to blockchain networks as a timestamping mechanism. The fundamental difference from AnchorKit is the complete absence of hardware attestation. The system records that a hash was submitted, but provides no evidence about the device that submitted it — the hash could be of a generated image submitted from a laptop. There is no equivalent of Android Key Attestation, verifiedBootState enforcement, or origin = GENERATED verification. AnchorKit's hardware-attestation layer elevates the provenance claim from "a hash was submitted at this time" to "a hash was captured on verified-boot hardware at this time."

§11 Scalability & Cost

The Merkle batching design decouples submission volume from on-chain cost. Regardless of whether 100 or 100,000 images are submitted on a given day, the nightly batch produces exactly one Solana transaction carrying one 32-byte Merkle root.

The off-chain submission pipeline is horizontally scalable. Rate limiting is enforced via DynamoDB atomic counters rather than per-process in-memory state; multiple concurrent backend workers each see the same rate limit state. The nonce atomicity guarantee (§5, step 4) holds across any number of concurrent workers.

The S3 archive and shard index provide O(1) hash-to-day lookup for proof generation. The shards/{p2}/{p4}.jsonl.gz index partitions the hash space by the first 4 hex characters, producing up to 65,536 possible shard files, each expected to contain only a small number of records at any realistic volume.

§12 Limitations & Future Work

12.1 Platform Scope: iOS

AnchorKit is currently Android-only. Apple's Secure Enclave (Apple, 2023) provides broadly comparable isolation properties to Android's StrongBox, but Apple does not expose a public API equivalent to Android Key Attestation for third-party verification of key hardware-binding and device boot state. Apple's App Attest API can prove an app binary is running on genuine Apple hardware but does not expose a verifiedBootState-equivalent field or key origin attestation. iOS support is in active development. A hybrid model combining App Attest with server-enforced timing constraints may provide a weaker but useful provenance signal; iOS proofs would require clearly differentiated trust-model presentation.

12.2 Time Granularity

AnchorKit's blockchain commitment establishes that a hash was submitted before midnight UTC on a given day. Within that day, precision is bounded by the five-minute nonce window plus the EXIF timestamp, which is not cryptographically authenticated. Future work: integrating RFC 3161 (Adams et al., 2001) trusted timestamp tokens would provide sub-second, independently auditable capture time with a trust chain separate from AnchorKit's infrastructure.

12.3 The Analogue Hole

AnchorKit cannot verify the provenance of the scene being photographed. Hardware attestation certifies the device and capture pipeline; it does not certify that a camera was pointed at a real physical scene. This is an inherent limitation of any camera-based provenance system. Research into depth sensing, multi-spectral capture, and temporal consistency detection may offer partial mitigations.

12.4 Image Content Verification

AnchorKit records that a specific hash was produced from a camera capture on an attested device; it makes no claim about the content or subject of the image. For use cases requiring content attestation, integration with zero-knowledge proof schemes allowing statements about image content to be proven without revealing the image is a promising direction for future work.

12.5 Key Continuity Across Reinstallations

A new key is generated per app installation. There is no mechanism to establish continuity of identity across reinstallations. Anchoring key generation events on-chain would be a useful addition for long-term monitoring use cases.

12.6 Google Attestation CA Dependency

The full chain of trust is rooted at Google's attestation CA. A compromise of Google's attestation signing keys would allow arbitrary certificate issuance. This risk is systemic to Android Key Attestation. Certificate transparency logging for attestation certificates — outside AnchorKit's control — would provide an additional detection layer.

12.7 Multi-Chain Anchoring

AnchorKit currently anchors to Solana only. Posting roots to multiple independent blockchains would reduce single-chain dependency. The Merkle root is chain-agnostic; extending the nightly batch to post to additional chains is architecturally straightforward.

§13 Conclusion

AnchorKit addresses a specific gap in the current landscape of digital media provenance. Content-based AI detection is a losing strategy in adversarial settings: the history of the field is a sequence of detectors rendered obsolete by the next generation of synthesis techniques. Hardware-rooted provenance provides a durable alternative — it makes the question of synthesis irrelevant at the protocol level, because synthetic images cannot produce valid hardware attestations.

The system achieves its properties through a composition of four independently well-understood primitives — Android Key Attestation, ECDSA binding over a time-limited nonce, Merkle tree commitment, and Solana blockchain anchoring — combined with a server-side validation pipeline that enforces all required attestation properties and correctly rejects both invalid field values and absent fields, preventing omission-bypass attacks.

The verification model is genuinely trustless: a PortableProof can be verified offline using only SHA-256 computation and a direct Solana RPC call. AnchorKit's servers can go offline without affecting the verifiability of any previously anchored proof.

The limitations of the system — the analogue hole, the iOS platform gap, the Google CA dependency, the daily (rather than sub-second) time granularity — are documented accurately as boundary conditions on its claims, not deficiencies in its implementation. Within its stated scope, AnchorKit provides a cryptographically sound, hardware-rooted, blockchain-anchored proof of a well-defined set of properties: that a given image hash was captured on genuine verified-boot Android hardware, with a key generated inside and never leaving a hardware-backed secure element, within a server-issued five-minute window, with the commitment anchored to a decentralised ledger accessible to any verifier, permanently.
